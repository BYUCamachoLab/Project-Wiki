# Dockerization Design — Project Wiki

**Date:** 2026-03-14
**Status:** Approved

## Overview

Containerize Project Wiki using Docker Compose with two services (app + MongoDB), using bind mounts into an existing `Project_Wiki_Data/` directory so that migration from an existing installation requires no data copying or `mongodump`/`mongorestore`. Caddy remains on the host machine and proxies to the containerized app.

## Architecture

```
Host machine
├── Caddy (host process)         ← TLS termination, reverse proxy
│     └── reverse_proxy localhost:8080
├── docker-compose.yml
│     ├── app (Flask/Waitress)   ← listens on 0.0.0.0:8080, exposed to host
│     └── mongo (MongoDB)        ← internal network only, not exposed to host
└── Project_Wiki_Data/
      ├── db/                    ← MongoDB data (bind mount → mongo container)
      └── uploads/               ← File uploads (bind mount → app container)
```

### Services

**`mongo` service**
- Image: `mongo:3.6` by default. Must match the major version of the existing MongoDB installation — run `mongod --version` on the host before setting up (see Migration Notes)
- Bind mount: `${DATA_DIR}/db` → `/data/db`
- Not exposed to host; reachable by `app` on the internal Docker network via hostname `mongo`
- Must be started with auth enabled: `command: ["mongod", "--auth"]` in `docker-compose.yml`
- Credentials must match those already embedded in the existing data files (see MongoDB auth note below)
- Healthcheck: `mongo --eval "db.adminCommand('ping')"` (the legacy `mongo` shell is the correct client for `mongo:3.6`; `mongosh` is not available in 3.x images)

**`app` service**
- Built from `Dockerfile` in repo root
- Base image: `python:3.9-slim` (compatible with the pinned dependencies: Flask 0.12.2, Werkzeug 0.12.2, MongoEngine 0.13.0)
- The Dockerfile uses `COPY . /app/` and `WORKDIR /app`, placing `config.py` at `/app/config.py`. This is required: `config.py` computes `UPLOAD_FOLDER` as one directory above its own location, so with `config.py` at `/app/config.py`, `basedir = /` and `UPLOAD_FOLDER = /Project_Wiki_Data/uploads`.
- Bind mount: `${DATA_DIR}/uploads` → `/Project_Wiki_Data/uploads`
- Port `8080` mapped to host (`127.0.0.1:8080:8080`) so Caddy can reach it
- Startup command: `python -c "from app import create_app; from waitress import serve; app = create_app(); serve(app, host='0.0.0.0', port=8080, threads=4)"`
  - `PW_run.py` is generated by `setup.sh` and is not committed to the repo; the Dockerfile uses a direct `CMD` instead
  - Uses `from app import create_app` directly rather than importing through `manage.py`, which avoids a dependency on `flask_script` at serve time
  - `waitress==1.0.2` (pinned in `macosx/requirements.txt`) does not support the `listen=` keyword; `host=` and `port=` must be used separately
  - Listen address is `0.0.0.0` (not `127.0.0.1`) to allow host port binding
- Dependencies installed from `macosx/requirements.txt` (the only platform-agnostic requirements file; Windows has an identical copy)
- Depends on `mongo` with `condition: service_healthy` so it waits for MongoDB to be ready

## Configuration

All configuration via a `.env` file (not committed; `.env.example` provided). Variable names match what `config.py` reads via `os.environ.get(...)`:

```
DATA_DIR=/absolute/path/to/Project_Wiki_Data

# MongoDB — must match the user already in the existing data files
DB_SERVICE=mongo
DB_PORT=27017
DB_NAME=admin
DB_USER=<existing mongodb username>
DB_PASS=<existing mongodb password>

# Flask
SECRET_KEY=<existing flask secret key>

# Super admin account
ADMIN_USERNAME=<existing admin username>
ADMIN_EMAIL=<existing admin email>
ADMIN_PASSWORD=<existing admin password>

# Email
MAIL_SERVER=smtp.googlemail.com
MAIL_PORT=587
MAIL_USERNAME=<email address>
MAIL_PASSWORD=<email password>
```

## Code Changes Required

These are the minimal changes to `config.py` needed for Docker compatibility:

```python
# Make MAIL_SERVER and MAIL_PORT env-configurable (were hardcoded constants)
MAIL_SERVER = os.environ.get('MAIL_SERVER', 'smtp.googlemail.com')
MAIL_PORT    = int(os.environ.get('MAIL_PORT', 587))

# Cast DB_PORT to int — env vars are strings; MongoEngine requires an integer
'port': int(os.environ.get('DB_PORT', 27017)),
```

The `DB_PORT` cast also fixes the secondary connection registrations in `app/__init__.py` (line 39), which read `config.MONGODB_SETTINGS['port']` directly — once the dict value is an int, both the primary and secondary connections work correctly.

## Migration from Existing Instance

### Before starting

1. **Verify your existing MongoDB version:** run `mongod --version` on the host. Update the `mongo` image tag in `docker-compose.yml` to match your major version (e.g., `mongo:3.6`, `mongo:4.4`). Also update the healthcheck command if using 4.x or later (replace `mongo` with `mongosh`).

2. **Collect your existing credentials:** the MongoDB username/password set during `setup.sh`, the Flask `SECRET_KEY`, and the admin account values. These go into `.env` verbatim — Docker does not re-create users in a non-empty data directory.

### Migration steps

1. **Stop existing services:** `cd macosx && bash stop.sh`
   - This stops MongoDB, the Python app, **and the host Caddy process** (`stop.sh` runs `killall caddy`). The wiki will be offline until Step 5.
2. **Populate `.env`** using `.env.example` as a template, filling in values from your existing setup
3. **Set `DATA_DIR`** in `.env` to the absolute path of the existing `Project_Wiki_Data/` directory
4. **Run `docker compose up -d`** — MongoDB finds its existing data in `db/`; the app finds uploads in `uploads/`
5. **Update the host Caddyfile** — add (or update) the site block for the wiki domain:
   ```caddyfile
   wiki.example.com {
       tls /path/to/cert.pem /path/to/key.pem
       reverse_proxy localhost:8080
   }
   ```
6. **Restart host Caddy:** `caddy reload` (or `sudo systemctl restart caddy` depending on your setup). The wiki is back online.

### MongoDB auth note

The official `mongo` Docker image only runs `MONGO_INITDB_*` initialization (creating the root user) when `/data/db` is **empty**. Since the bind-mounted `db/` already contains existing data files, those variables are ignored. MongoDB starts with `--auth` (via the `command` override in `docker-compose.yml`) and expects the user credentials already present in the data files. The `.env` values for `DB_USER` / `DB_PASS` must exactly match the credentials set during the original `setup.sh` run.

## Caddy Integration

The user maintains an existing Caddy installation on the host. A new site block is added to their existing Caddyfile (Caddy is not run inside Docker):

```caddyfile
wiki.example.com {
    tls /path/to/cert.pem /path/to/key.pem
    reverse_proxy localhost:8080
}
```

Caddy handles TLS termination with user-provided certificates. The app container only speaks plain HTTP internally on port 8080.

## Out of Scope

- Automated `mongodump` backups (host-level backup of `Project_Wiki_Data/` covers both DB and uploads)
- MongoDB major-version upgrades (containerize with matching version first, upgrade separately)
- Multi-replica or high-availability MongoDB
- Kubernetes / Swarm deployment
- Development vs. production Docker profiles (single compose file for production use)
